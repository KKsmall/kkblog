HashMap类

  1. HashMap类定义

    HashMap允许空值和空键。(HashMap类大致相当于Hashtable，只是它是不同步的，并且允许空值。)这个类不能保证映射的顺序;特别是，它不能保证顺序随时间保持不变。这个实现为基本操作(get和put)提供了恒定时间的性能，HashMap底层是通过数组(Node结点类型的数组)+链表(每个Node结点有指向下一个结点的指针从而形成链表)的形式实现，假设哈希函数将元素正确地分散在bucket(也就是Node数组)中。对集合视图的迭代需要的时间与HashMap实例的“容量”(bucket的数量)加上它的大小(键-值映射的数量)成正比。因此，如果迭代性能很重要，那么不要将初始容量设置得太高(或负载系数设置得太低)，这一点非常重要。
    HashMap的实例有两个影响其性能的参数:初始容量和装载因子。容量是哈希表中的桶数，初始容量就是创建哈希表时的容量。装载因子是一种度量方法，用来衡量在自动增加哈希表的容量之前，哈希表允许的满度。当哈希表中的条目数超过负载因子和当前容量的乘积时，哈希表将被重新哈希(即重新构建内部数据结构)，这样哈希表的桶数大约是原来的两倍。

  2. HashMap中的几个属性

    DEFAULT_INITIAL_CAPACITY默认初始容量必须是2的幂次方,原因是为了保证每个元素能均匀的散落在数组位置中，在put()中有个取模的操作就是为了这个，具体看后面的解析
    DEFAULT_LOAD_FACTOR默认负载因子设置为0.75是取得的一个适中的值，因为负载因子是表示疏密的程度，如果太大了说明能容忍元素的数量很多，就可能造成链表的长度过长，太小了就使得利用率不够
    TREEIFY_THRESHOLD和MIN_TREEIFY_CAPACITY共同决定数组某个下标下的结点链表是否要转成红黑树，如果链表长度大于TREEIFY_THRESHOLD(8)并且数组容量大于MIN_TREEIFY_CAPACITY(64)，则将链表转成红黑树

  5. 其他方法

    1. static final int hash(Object key)

      hash方法用来确定元素散落在数组中的下标位置，并且分布较均匀

    2. static final int tableSizeFor(int cap)

      该方法返回指定容量的2倍幂，使HashMap的容量永远保持2的幂次方,比如传的值是10,2的3次方是8小于10于是就2的4次方 结果为16，当容量超过16就要调整大小了

    3. get(Object key)

      获取key映射的值

    4. containsKey(Object key)

      判断HashMap中是否包含指定key,只要getNode方法能找到不为Null的结点说明就包含

    5. containsValue(Object value)

      判断map中是否包含指定的值

    6. remove(Object key)

      移除对应key的结点

    7. clear()

      清除map中的所有映射

    8. replace(K key, V oldValue, V newValue)

      替换key中的旧值,判断对应结点中的值是否和oldValue相同，相同则替换新值

    9. replace(K key, V value)

      替换值但不做旧值的判断直接替换，并返回旧值
